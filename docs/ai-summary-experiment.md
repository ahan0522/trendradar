# AI 摘要手動實驗 v1

本文件用目前線上真實資料手動模擬 LLM 摘要結果，目標是在不花 API 費用前，先確認 TrendRadar 的摘要格式與資訊密度。

## 測試目的

TrendRadar 的詳情頁不應只是列新聞標題，而是讓使用者快速知道：

- 這篇來源說了什麼事。
- 它和主題的關係是什麼。
- 有沒有事實查證價值或只是延伸背景。
- 是否需要點原文。

## 建議 Prompt

```text
你是 TrendRadar 的新聞整理助手。

任務：
把多篇來源文章整理成「快速理解」格式。不要複製新聞標題，不要使用聳動語氣，不要自行補不存在的事實。

輸入：
- topicTitle: 主題名稱
- category: 分類
- articles: 來源文章陣列，每篇包含 title、description、sourceName

輸出請使用繁體中文 JSON：
{
  "topicSummary": "用 2-3 句說明這個主題目前發生什麼事",
  "keyPoints": [
    "不重複的重點 1",
    "不重複的重點 2",
    "不重複的重點 3"
  ],
  "sourceBriefs": [
    {
      "sourceRole": "主要事件 | 延伸背景 | 市場反應 | 查證來源 | 待確認",
      "brief": "這篇來源用一句話說明發生什麼事",
      "whyItMatters": "它對主題的重要性",
      "confidence": "high | medium | low"
    }
  ],
  "openQuestions": [
    "目前仍不確定或後續要追蹤的問題"
  ]
}

規則：
- 每篇 brief 以 35-60 字為主。
- 如果 description 只是標題集合，要整合，不要照抄。
- 如果來源是人物、背景、評論或市場反應，要明確標成延伸背景或市場反應。
- 熱門不等於正確；涉及爆料、推測、估值、軍事風險時 confidence 不要直接給 high。
```

## 實驗結論

LLM 摘要最有價值的地方不是把標題改寫，而是判斷每篇來源的角色：

- 主要事件
- 延伸背景
- 市場反應
- 查證來源
- 待確認

這比目前 rule-based quickSummary 更適合 TrendRadar，因為同一個主題裡常混有產業事件、人物背景、市場評論、風險爆料。使用者需要的是「這篇在主題裡扮演什麼角色」。

## 建議實作方向

先不要一次把所有文章都送 AI。建議：

1. 每個主題先挑 6-8 篇去重後代表來源。
2. 每個主題只呼叫一次 LLM。
3. LLM 回傳 topicSummary、keyPoints、sourceBriefs、openQuestions。
4. 結果寫入 topics.summary、topics.bullets，sourceBriefs 後續可新增欄位或先存在衍生 JSON。
5. 每天 cron 1-2 次即可，先控制成本。

如果暫時不接 API，也可以先把 prompt 放在手動流程中，用 ChatGPT 測不同主題的摘要品質。
